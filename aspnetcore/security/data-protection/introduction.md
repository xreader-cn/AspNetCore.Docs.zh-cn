## <a name="problem-statement"></a><span data-ttu-id="b1b43-109">问题陈述</span><span class="sxs-lookup"><span data-stu-id="b1b43-109">Problem statement</span></span>

<span data-ttu-id="b1b43-110">整体问题声明可在一个句子中简单地表述：我需要保存可信信息供以后检索，但不信任持久性机制。</span><span class="sxs-lookup"><span data-stu-id="b1b43-110">The overall problem statement can be succinctly stated in a single sentence: I need to persist trusted information for later retrieval, but I don't trust the persistence mechanism.</span></span> <span data-ttu-id="b1b43-111">在 web 术语中，这可能被编写为 "我需要通过不受信任的客户端往返的受信任状态。"</span><span class="sxs-lookup"><span data-stu-id="b1b43-111">In web terms, this might be written as "I need to round-trip trusted state via an untrusted client."</span></span>

<span data-ttu-id="b1b43-112">此规范的典型示例是身份验证 cookie 或持有者令牌。</span><span class="sxs-lookup"><span data-stu-id="b1b43-112">The canonical example of this is an authentication cookie or bearer token.</span></span> <span data-ttu-id="b1b43-113">服务器生成一个 "I Groot" 和 "xyz 权限" 令牌并将其交给客户端。</span><span class="sxs-lookup"><span data-stu-id="b1b43-113">The server generates an "I am Groot and have xyz permissions" token and hands it to the client.</span></span> <span data-ttu-id="b1b43-114">在将来的某个日期，客户端会将该令牌提供给服务器，但服务器需要某种形式的保证，那就是客户端未伪造令牌。</span><span class="sxs-lookup"><span data-stu-id="b1b43-114">At some future date the client will present that token back to the server, but the server needs some kind of assurance that the client hasn't forged the token.</span></span> <span data-ttu-id="b1b43-115">因此，第一要求：真实性（也称为</span><span class="sxs-lookup"><span data-stu-id="b1b43-115">Thus the first requirement: authenticity (a.k.a.</span></span> <span data-ttu-id="b1b43-116">完整性，防篡改。</span><span class="sxs-lookup"><span data-stu-id="b1b43-116">integrity, tamper-proofing).</span></span>

<span data-ttu-id="b1b43-117">由于持久状态受服务器信任，因此我们预计此状态可能包含特定于操作环境的信息。</span><span class="sxs-lookup"><span data-stu-id="b1b43-117">Since the persisted state is trusted by the server, we anticipate that this state might contain information that's specific to the operating environment.</span></span> <span data-ttu-id="b1b43-118">这可能是文件路径、权限、句柄或其他间接引用的形式，也可能是其他特定于服务器的数据的形式。</span><span class="sxs-lookup"><span data-stu-id="b1b43-118">This could be in the form of a file path, a permission, a handle or other indirect reference, or some other piece of server-specific data.</span></span> <span data-ttu-id="b1b43-119">此类信息通常不会泄露给不受信任的客户端。</span><span class="sxs-lookup"><span data-stu-id="b1b43-119">Such information should generally not be disclosed to an untrusted client.</span></span> <span data-ttu-id="b1b43-120">因此，第二个要求：保密性。</span><span class="sxs-lookup"><span data-stu-id="b1b43-120">Thus the second requirement: confidentiality.</span></span>

<span data-ttu-id="b1b43-121">最后，由于新式应用程序是组件化的，我们看到的是，单个组件需要利用此系统，而不考虑系统中的其他组件。</span><span class="sxs-lookup"><span data-stu-id="b1b43-121">Finally, since modern applications are componentized, what we've seen is that individual components will want to take advantage of this system without regard to other components in the system.</span></span> <span data-ttu-id="b1b43-122">例如，如果持有者令牌组件正在使用此堆栈，则它应在不干扰可能使用同一堆栈的 CSRF 机制的情况下运行。</span><span class="sxs-lookup"><span data-stu-id="b1b43-122">For instance, if a bearer token component is using this stack, it should operate without interference from an anti-CSRF mechanism that might also be using the same stack.</span></span> <span data-ttu-id="b1b43-123">因此，最终要求：隔离。</span><span class="sxs-lookup"><span data-stu-id="b1b43-123">Thus the final requirement: isolation.</span></span>

<span data-ttu-id="b1b43-124">我们可以提供进一步的约束来缩小需求的范围。</span><span class="sxs-lookup"><span data-stu-id="b1b43-124">We can provide further constraints in order to narrow the scope of our requirements.</span></span> <span data-ttu-id="b1b43-125">假设在 cryptosystem 内运行的所有服务都是同等信任的，并且无需在我们的直接控制下生成或使用服务外部的数据。</span><span class="sxs-lookup"><span data-stu-id="b1b43-125">We assume that all services operating within the cryptosystem are equally trusted and that the data doesn't need to be generated or consumed outside of the services under our direct control.</span></span> <span data-ttu-id="b1b43-126">此外，我们还需要尽可能快地执行操作，因为每次向 web 服务发出的请求可能会经过 cryptosystem 一次或多次。</span><span class="sxs-lookup"><span data-stu-id="b1b43-126">Furthermore, we require that operations are as fast as possible since each request to the web service might go through the cryptosystem one or more times.</span></span> <span data-ttu-id="b1b43-127">这使得对称加密非常适合我们的方案，并且我们可以为非对称加密提供折扣，直到需要这样的时间。</span><span class="sxs-lookup"><span data-stu-id="b1b43-127">This makes symmetric cryptography ideal for our scenario, and we can discount asymmetric cryptography until such a time that it's needed.</span></span>

## <a name="design-philosophy"></a><span data-ttu-id="b1b43-128">设计理念</span><span class="sxs-lookup"><span data-stu-id="b1b43-128">Design philosophy</span></span>

<span data-ttu-id="b1b43-129">首先，我们要找出现有堆栈的问题。</span><span class="sxs-lookup"><span data-stu-id="b1b43-129">We started by identifying problems with the existing stack.</span></span> <span data-ttu-id="b1b43-130">完成这一工作后，我们调查了现有解决方案的前景，并最终找不到现有解决方案。</span><span class="sxs-lookup"><span data-stu-id="b1b43-130">Once we had that, we surveyed the landscape of existing solutions and concluded that no existing solution quite had the capabilities we sought.</span></span> <span data-ttu-id="b1b43-131">然后，基于多个指导原则对解决方案进行了设计。</span><span class="sxs-lookup"><span data-stu-id="b1b43-131">We then engineered a solution based on several guiding principles.</span></span>
